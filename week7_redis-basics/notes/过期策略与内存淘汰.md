## 一、Redis 过期策略

### 1.1 为什么需要过期策略？

| 问题 | 解决方案 |
|------|----------|
| 临时数据（验证码、Session）| 设置过期时间，自动清理 |
| 内存有限 | 淘汰不常用的 key |

### 1.2 两种过期策略

| 策略 | 原理 | 触发时机 | CPU 占用 | 内存释放 |
|------|------|----------|----------|----------|
| **惰性删除** | 访问时检查是否过期 | 读/写 key 时 | 低 | 不及时 |
| **定期删除** | 定时抽取批量检查 | 每秒 10 次 | 中 | 及时 |

**结论**：两者配合使用，兼顾性能和内存。

二、Redis 内存淘汰策略
2.1 为什么需要淘汰？
当内存达到 maxmemory 限制时，新写入操作会失败。淘汰策略决定删哪些 key。

2.2 配置方式
bash
# 命令行动态配置（推荐）
CONFIG SET maxmemory 256mb
CONFIG SET maxmemory-policy allkeys-lru

# 永久生效
CONFIG REWRITE
conf
# redis.conf 配置
maxmemory 256mb
maxmemory-policy allkeys-lru
三、8 种淘汰策略详解
策略	范围	算法	说明
noeviction	-	-	不淘汰，内存满时写入报错
allkeys-lru	所有 key	LRU	淘汰最近最少使用的
volatile-lru	有过期时间的 key	LRU	淘汰最近最少使用的
allkeys-random	所有 key	随机	随机淘汰
volatile-random	有过期时间的 key	随机	随机淘汰
volatile-ttl	有过期时间的 key	TTL	淘汰剩余时间最短的
allkeys-lfu（4.0+）	所有 key	LFU	淘汰最不经常使用的
volatile-lfu（4.0+）	有过期时间的 key	LFU	淘汰最不经常使用的
四、常用策略（重点）
1. allkeys-lru（最常用）
适用场景：通用缓存、热点数据

原理：淘汰最久没被访问的 key，保留热点数据。

bash
CONFIG SET maxmemory-policy allkeys-lru
2. noeviction（数据安全）
适用场景：不允许丢失数据的场景

bash
CONFIG SET maxmemory-policy noeviction
3. volatile-ttl（临时数据）
适用场景：验证码、Session、临时 Token

bash
CONFIG SET maxmemory-policy volatile-ttl
4. allkeys-lfu（防刷场景）
适用场景：访问频率差异大的场景

bash
CONFIG SET maxmemory-policy allkeys-lfu
五、策略选择指南
场景	推荐策略	原因
通用缓存（热点数据）	allkeys-lru	保留热点，淘汰冷门
验证码/Session	volatile-ttl	优先淘汰快过期的
不允许丢失数据	noeviction	宁可报错也不丢数据
防刷、统计场景	allkeys-lfu	淘汰访问次数少的
循环写入日志	allkeys-random	不在乎淘汰谁
六、LRU vs LFU
算法	全称	原理	示例
LRU	Least Recently Used	淘汰最久没访问的	网页缓存
LFU	Least Frequently Used	淘汰访问次数最少的	热点统计、防刷
text
LRU 示例：访问顺序：A, B, C, A, D → 淘汰 B
LFU 示例：访问次数：A(10次), B(5次), C(1次) → 淘汰 C
