# 索引失效场景清单

## 一、索引失效的核心原因

**索引失效的本质：MySQL 无法利用 B+ 树的有序性来快速定位数据。**

## 二、10 种索引失效场景

### 1. 左模糊查询（LIKE '%xxx'）

```sql
-- ❌ 失效
SELECT * FROM user WHERE name LIKE '%三';

-- ✅ 有效
SELECT * FROM user WHERE name LIKE '张%';
原因：B+ 树按前缀排序，左模糊无法定位起点。

2. 对索引列使用函数
sql
-- ❌ 失效
SELECT * FROM user WHERE LEFT(name, 2) = '张三';

-- ✅ 有效
SELECT * FROM user WHERE name LIKE '张三%';
原因：索引存的是原值，不是函数计算结果。

3. 隐式类型转换
sql
-- ❌ 失效（phone 是 VARCHAR）
SELECT * FROM user WHERE phone = 13800138000;

-- ✅ 有效
SELECT * FROM user WHERE phone = '13800138000';
原因：MySQL 会进行类型转换，相当于对索引列用了函数。

4. 不满足最左前缀法则
sql
-- 联合索引 (name, age)
-- ❌ 失效
SELECT * FROM user WHERE age = 20;

-- ✅ 有效
SELECT * FROM user WHERE name = '张三' AND age = 20;
原因：跳过了最左列，无法利用索引的有序性。

5. 使用 OR 条件
sql
-- ❌ 可能失效
SELECT * FROM user WHERE name = '张三' OR age = 20;

-- ✅ 优化：用 UNION
SELECT * FROM user WHERE name = '张三'
UNION
SELECT * FROM user WHERE age = 20;
原因：OR 两边可能走不同索引，优化器可能选择全表扫描。

6. 使用 != 或 <>
sql
-- ❌ 失效
SELECT * FROM user WHERE name != '张三';
原因：负向查询范围太大，MySQL 认为全表扫描更优。

7. IS NULL / IS NOT NULL
sql
-- 🟡 可能失效（取决于数据分布）
SELECT * FROM user WHERE name IS NULL;
8. 索引列参与计算
sql
-- ❌ 失效
SELECT * FROM user WHERE age + 1 = 21;

-- ✅ 有效
SELECT * FROM user WHERE age = 20;
9. 使用 NOT IN
sql
-- 🟡 可能失效
SELECT * FROM user WHERE name NOT IN ('张三', '李四');
10. 范围查询后面的列失效
sql
-- 联合索引 (name, age, city)
-- ❌ city 用不到索引
SELECT * FROM user 
WHERE name = '张三' AND age > 20 AND city = '北京';
规则：范围查询（>、<、between）后面的索引列都会失效。
