# 聚合查询优化

## 一、什么是聚合查询？

```sql
-- 统计订单总额
SELECT SUM(amount) FROM `order`;

-- 统计每天订单数
SELECT DATE(create_time), COUNT(*) FROM `order` GROUP BY DATE(create_time);
问题：大表上做聚合，需要全表扫描，千万级数据时非常慢。

二、为什么慢？
原因	说明
全表扫描	需要读取所有数据行
临时表	GROUP BY 需要创建临时表排序
磁盘 I/O	数据量大，内存装不下，写磁盘
三、优化方案
方案	原理	适用场景
汇总表	提前算好结果，存起来	固定维度的统计报表
临时表	分批处理，减少锁	一次性复杂查询
覆盖索引	索引包含查询字段	简单聚合
四、方案1：汇总表（推荐）
建表
sql
-- 汇总表（小表）
CREATE TABLE order_stats (
    `stat_date` DATE PRIMARY KEY,
    `order_count` INT,
    `total_amount` DECIMAL(10,2),
    `update_time` DATETIME
);
定时更新（每天凌晨执行）
sql
INSERT INTO order_stats (stat_date, order_count, total_amount, update_time)
SELECT 
    DATE(create_time),
    COUNT(*),
    SUM(amount),
    NOW()
FROM `order`
WHERE DATE(create_time) = CURDATE() - INTERVAL 1 DAY
GROUP BY DATE(create_time)
ON DUPLICATE KEY UPDATE
    order_count = VALUES(order_count),
    total_amount = VALUES(total_amount),
    update_time = VALUES(update_time);
查询（毫秒级）
sql
-- 原来：扫全表（慢）
SELECT SUM(amount) FROM `order`;

-- 优化后：读汇总表（快）
SELECT SUM(total_amount) FROM order_stats;
五、方案2：临时表 + 批量处理
sql
-- 1. 创建临时表
CREATE TEMPORARY TABLE tmp_order_stats (
    `user_id` INT,
    `total_amount` DECIMAL(10,2),
    PRIMARY KEY (`user_id`)
);

-- 2. 分批插入
INSERT INTO tmp_order_stats (user_id, total_amount)
SELECT user_id, SUM(amount) FROM `order` 
WHERE id BETWEEN 1 AND 100000
GROUP BY user_id
ON DUPLICATE KEY UPDATE total_amount = total_amount + VALUES(total_amount);

-- 3. 查询临时表
SELECT * FROM tmp_order_stats ORDER BY total_amount DESC LIMIT 10;
六、性能对比（实验数据）
查询方式	耗时	扫描行数
直接聚合	68.8ms	全表（数千行）
汇总表（7天）	<1ms	7 行
提升约 70 倍！

七、适用场景判断
场景	是否用汇总表
每日报表	✅ 适合
实时大屏	❌ 不适合（有延迟）
历史数据分析	✅ 适合
多维度交叉统计	🟡 需要多个汇总表
八、面试速答
问题	答案
大表 count 慢怎么办？	用汇总表提前统计
汇总表有什么缺点？	实时性差，需要定时更新
多维度统计怎么办？	建多个汇总表或用 OLAP
