# 分页查询优化

## 一、两种分页方式对比

| 对比项 | limit 分页 | 游标分页 |
|--------|-----------|----------|
| 跳页 | ✅ 能 | ❌ 不能 |
| 下一页 | ✅ 能 | ✅ 能 |
| 深度分页性能 | ❌ 越往后越慢 | ✅ 快 |
| 原理 | 扫描所有数据，跳过偏移量 | `WHERE id > last_id` |

---

## 二、Explain 验证结果

### limit 分页

```sql
EXPLAIN SELECT * FROM user ORDER BY id LIMIT 499990, 10;
type: index（全索引扫描）

rows: 625（实际扫描 50 万行）❌

游标分页
sql
EXPLAIN SELECT * FROM user WHERE id > 499990 ORDER BY id LIMIT 10;
type: range（范围查询）✅

rows: 1 ✅

三、为什么游标分页不能跳页？
id 不连续（删除、插入失败、分布式 id 导致）

text
id 列表：1, 2, 5, 8, 9, 10...
第 2 页（每页 2 条）应该是 id=5 开始，但 id=3,4 不存在
无法用公式计算 id 位置
四、解决方案：软删除 + 游标分页
sql
-- 软删除（不真删，只标记）
UPDATE user SET is_deleted = 1 WHERE id = 500;

-- 游标分页（id 连续，可以算位置）
SELECT * FROM user WHERE is_deleted = 0 AND id > ? LIMIT 10;
优势：

✅ id 连续 → 可以用游标跳页

✅ 可恢复数据

✅ 性能好

代价：

存储空间占用

查询需要过滤 is_deleted = 0

五、适用场景
场景	推荐方案
后台管理（数据量小）	limit 分页
App 信息流	游标分页
订单/用户表	软删除 + 游标
日志数据	限制最大页码
六、一句话总结
limit 适合跳页但深度分页慢，游标快但不能跳页。软删除 + 游标可以同时解决这两个问题。
