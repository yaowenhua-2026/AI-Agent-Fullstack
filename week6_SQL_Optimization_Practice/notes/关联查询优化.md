# 关联查询优化

## 一、三种 Join 算法

| 算法 | 原理 | 复杂度 | MySQL 版本 |
|------|------|--------|------------|
| **NLJ（嵌套循环）** | 驱动表每行去被驱动表匹配 | O(N×M) | 所有版本 |
| **BNL（块嵌套循环）** | 驱动表分批，放入内存匹配 | O(N×M/R) | 5.6+ |
| **Hash Join** | 建立哈希表匹配 | O(N+M) | 8.0.18+ |

---

## 二、实验验证

### 不加索引

```sql
EXPLAIN SELECT * FROM `order` o JOIN `user` u ON o.user_id = u.id;
表	type	key	rows
order	ALL	NULL	100078
user	eq_ref	PRIMARY	1
问题：order 表全表扫描 ❌

加索引
sql
ALTER TABLE `order` ADD INDEX idx_user_id (user_id);
表	type	key	rows
user	ALL	PRIMARY	10054
order	ref	idx_user_id	10
效果：order 从全表扫描 → 索引查找 ✅

三、性能对比
状态	被驱动表 type	被驱动表 rows	效果
不加索引	ALL	100078	全表扫描 ❌
加索引	ref	10	索引查找 ✅
四、Join 优化清单
优化策略	说明
✅ 被驱动表加索引	最关键！让 NLJ 能用索引
✅ 小表驱动大表	减少驱动表遍历次数
✅ 只查需要的列	减少数据传输
✅ 用 EXPLAIN 分析	找出慢的原因
✅ 避免超过 3 张表 join	考虑拆分
🟡 升级 MySQL 8.0	用 Hash Join
五、面试速答
问题	答案
join 慢怎么办？	被驱动表加索引
为什么加索引能优化？	让 NLJ 从全表扫描变成索引查找
小表驱动大表为什么对？	减少驱动表遍历次数
六、一句话总结
join 优化的核心：被驱动表的 join 字段加索引，让 MySQL 用小表驱动大表。
