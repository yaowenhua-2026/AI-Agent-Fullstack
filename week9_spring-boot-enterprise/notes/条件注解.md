# 条件注解（@Conditional）

## 学习时间
2026-05-22

---

## 一、核心概念

**条件注解的作用：** 让 Spring 容器根据特定条件（Bean 是否存在、配置属性值、类是否存在等）**决定是否创建某个 Bean 或加载整个配置类**。

这是 Spring Boot 自动配置的基石，也是自定义 Starter 的必备技术。

---

## 二、三大核心条件注解

| 注解 | 判断依据 | 典型用途 | 记忆口诀 |
|------|----------|----------|----------|
| `@ConditionalOnMissingBean` | 容器中**没有**某类型的 Bean | 提供可被用户覆盖的默认实现 | "你不来，我来" |
| `@ConditionalOnProperty` | 配置文件中的属性值 | 功能开关、环境适配 | "听配置的" |
| `@ConditionalOnClass` | 类路径中**存在**某个类 | 根据依赖自动适配实现 | "有你才行" |

---

## 三、场景1：@ConditionalOnMissingBean

### 解决的问题

提供一个**默认实现**，但如果用户自己定义了，就优先用用户的。

### 代码模板

```java
@AutoConfiguration
@ConditionalOnMissingBean(SmsSender.class)  // 容器中没有 SmsSender 时才生效
public class SmsAutoConfiguration {
    @Bean
    public SmsSender defaultSmsSender() {
        return new DefaultSmsSender();  // 默认实现
    }
}
用户使用
java
// 情况1：用户什么都不做 → 自动获得 DefaultSmsSender
// 情况2：用户自己定义
@Bean
public SmsSender aliyunSmsSender() {
    return new AliyunSmsSender();  // 覆盖默认实现
}
执行逻辑
text
容器中有 SmsSender 吗？
├── 没有 → 条件满足 ✅ → 创建默认 Bean
└── 有 → 条件不满足 ❌ → 跳过
四、场景2：@ConditionalOnProperty
解决的问题
让用户通过配置文件控制功能开关。

代码模板
java
@AutoConfiguration
@ConditionalOnProperty(
    prefix = "monitor",      // 配置前缀
    name = "enabled",        // 配置名称 → 组合成 monitor.enabled
    havingValue = "true",    // 期望的值
    matchIfMissing = false   // 没配置时默认不创建（安全优先）
)
public class MonitorAutoConfiguration {
    @Bean
    public PerformanceMonitor performanceMonitor() {
        return new PerformanceMonitor();
    }
}
用户使用
yaml
# application.yml
monitor:
  enabled: true   # 开启监控
执行逻辑
text
读取 monitor.enabled 配置
├── 值为 "true" → 条件满足 ✅ → 创建 Bean
├── 值为 "false" → 条件不满足 ❌ → 不创建
└── 没配置 → matchIfMissing=false → 不创建
常用参数组合
场景	配置
必须开启才生效	havingValue = "true", matchIfMissing = false
默认开启	havingValue = "true", matchIfMissing = true
只要配置了就生效	name = "feature.enabled"（不指定 havingValue）
五、场景3：@ConditionalOnClass
解决的问题
根据用户引入的依赖，自动切换实现（有 Redis 用 Redis，没有就用本地缓存）。

代码模板
java
@AutoConfiguration
public class CacheAutoConfiguration {

    // 情况1：有 Redis 依赖
    @Configuration
    @ConditionalOnClass(RedisTemplate.class)
    static class RedisCacheConfig {
        @Bean
        @ConditionalOnMissingBean
        public CacheService redisCacheService(RedisTemplate<String, Object> redisTemplate) {
            return new RedisCacheService(redisTemplate);
        }
    }

    // 情况2：没有 Redis 依赖（降级）
    @Configuration
    @ConditionalOnMissingClass("org.springframework.data.redis.core.RedisTemplate")
    static class LocalCacheConfig {
        @Bean
        @ConditionalOnMissingBean
        public CacheService localCacheService() {
            return new LocalCacheService();
        }
    }
}
用户使用
xml
<!-- 引入 Redis 依赖 → 自动用 Redis 缓存 -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis</artifactId>
</dependency>
执行逻辑
text
类路径中有 RedisTemplate 吗？
├── 有 → 条件满足 ✅ → 创建 RedisCacheService
└── 没有 → @ConditionalOnMissingClass 满足 → 创建 LocalCacheService
六、AutoConfiguration.imports 注册文件
文件路径
text
src/main/resources/META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
文件内容（每行一个自动配置类的全限定名）
text
com.example.autoconfigure.SmsAutoConfiguration

作用
告诉 Spring Boot："这里有我的自动配置类，请考虑是否加载它们"。

七、三种场景对比一览表
维度	场景1	场景2	场景3
条件注解	@ConditionalOnMissingBean	@ConditionalOnProperty	@ConditionalOnClass
判断依据	容器中是否有某类型 Bean	配置文件中的属性值	类路径中是否有某类
用户控制方式	自己定义 Bean	修改配置文件	引入/不引入依赖
典型例子	短信默认实现	性能监控开关	Redis/本地缓存切换
设计模式	可覆盖的默认实现	功能开关	自动适配
八、组合使用示例
实际开发中，经常组合使用多个条件：

java
@AutoConfiguration
@ConditionalOnClass(SqlSessionFactory.class)  // 有 MyBatis
@ConditionalOnProperty(prefix = "mybatis", name = "enabled", havingValue = "true")
public class MybatisAutoConfig {
    @Bean
    @ConditionalOnMissingBean
    public SqlSessionFactory sqlSessionFactory(DataSource dataSource) {
        // 只有在：有 MyBatis + 配置开启 + 用户没自定义 → 才创建
        return new MybatisSqlSessionFactoryBuilder().build(dataSource);
    }
}
九、面试常问
Q1：@ConditionalOnMissingBean 和 @ConditionalOnClass 有什么区别？
对比	@ConditionalOnMissingBean	@ConditionalOnClass
判断对象	Spring 容器中的 Bean	类路径中的类
判断时机	运行时（容器已启动）	类加载时
典型用途	可覆盖的默认实现	根据依赖适配
Q2：matchIfMissing 参数的作用？
控制当配置文件没有这个属性时，条件是否满足。

true → 没配置也满足（默认开启）

false → 没配置不满足（默认关闭，推荐）

Q3：为什么要用内部 @Configuration 类？
因为正反两个条件（@ConditionalOnClass 和 @ConditionalOnMissingClass）不能写在同一个类上，分开写让 Spring 根据条件选择加载哪一个。

十、总结
条件注解让 Spring Boot 变得"智能"：根据环境、配置、依赖自动决定创建哪些 Bean，实现了"开箱即用，用户优先"的设计理念。
